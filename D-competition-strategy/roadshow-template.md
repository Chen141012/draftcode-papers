# 路演模板（6分钟）

> 比赛评分 30% 权重。6 分钟讲清楚你做了什么、为什么这么做、结果如何。
> 结构固定，照着填内容就行。

## 幻灯片结构（6 张核心 + 2 张备用）

```
Slide 1 ─ 开场（30秒）
Slide 2 ─ 挑战（1分钟）
Slide 3 ─ 方案（2分钟）← 技术核心
Slide 4 ─ 演示（1分30秒）
Slide 5 ─ 亮点（30秒）
Slide 6 ─ 收尾（30秒）

[备用] Slide 7 ─ 技术细节深挖
[备用] Slide 8 ─ 数据来源/方法论
```

---

### Slide 1：开场（0:00-0:30）

**一句话定调：**
> "我们用 AWS Bedrock + XGBoost，基于 26 年历史数据，构建了一个 NBA 选秀预测 Agent。"

**不要写：**
- 不要自我介绍（评委手里有名单）
- 不要感谢主办方
- 不要"大家好我们来自..."

**建议画面：**
- 1 张 MVP 截图（Web Agent 界面的主页）

---

### Slide 2：核心挑战（0:30-1:30）

**讲清楚问题就够了，不用煽情：**

```
Challenge 1: 26 年数据 → 30 支球队 → 60 个顺位
Challenge 2: 数据稀疏性（很多球员大学数据不完全）
Challenge 3: 24 小时内完成 数据清洗→建模→Agent→UI
```

**评委想听：** 你们理解这个问题的难度吗？

**建议画面：**
- 数据规模简表（如：X 条记录，Y 个特征）
- 挑战 3 点列表

---

### Slide 3：技术方案（1:30-3:30）⭐ 核心

**两句话概括 + 一张图：**

```
数据 (26年NBA历史数据)
    ↓
特征工程 (体测/大学数据/效率指标 → 标准化特征矩阵)
    ↓
XGBoost 预测模型 (训练 → 30队预测 → 置信度)
    ↓
Agent 层 (Bedrock Tool Use → 分析/预测/可视化)
    ↓
Web UI (交互式选秀板)
```

**评委想听：**
- 用了什么技术
- 为什么选这些技术（不用讲得太技术细节）
- 架构的合理性

**建议画面：**
- 架构图（数据处理 + 模型 + Agent + UI 四层）
- 模型关键指标（MAE, 最准的 3 个预测）

---

### Slide 4：现场演示（3:30-5:00）

**操作流程：**
```
1. 打开 Web Agent 界面（30秒）
2. 输入查询："预测马刺队#4顺位"（30秒）
3. Agent 展示推理过程 + 结果（30秒）
4. 再查一个验证泛化能力（30秒）
```

**⚠️ 必做准备：**
- [ ] 准备录屏备份（现场网络/Agent 可能崩）
- [ ] 至少测试 5 次演示场景
- [ ] 备选问题（万一第一个挂了）
- [ ] 关掉无关窗口、通知

**评委想听：** 你们的 Agent 真的能用吗？

---

### Slide 5：亮点（5:00-5:30）

**选 3 个最有说服力的：**

1. **命中率** — "我们预测的 30 队中 XX 队与实际一致"
2. **技术亮点** — "XGBoost + 特征重要性分析识别出 XXX 是最关键因子"
3. **Agent 能力** — "只需自然语言查询，就能获得完整选秀分析"

**或者如果预测不准也诚实：**
> "命中率 XX%，但我们的 Agent 框架和数据管道可以直接复用到下一届比赛。"

---

### Slide 6：收尾（5:30-6:00）

```
✅ 26 年数据 → 30 队预测 → Web Agent 界面
✅ XGBoost 模型 + AWS Bedrock Agent
✅ 可交互、可解释的选秀预测系统

谢谢！欢迎提问。
```

## 应对评委提问

| 可能的提问 | 回答方向 |
|-----------|---------|
| 你们用了哪些特征？最关键的因子是什么？ | 见 A3-feature-engineering，答 3 个最重要的特征 |
| 模型准确率多少？ | 见训练评估指标，诚实回答 MAE |
| 为什么选 XGBoost 而不是深度学习？ | 数据量小 + 可解释性 + 24h 时间限制 |
| Agent 和其他方案有什么区别？ | 自然语言交互 + 可解释推理过程 |
| 如果时间多一天你会做什么？ | 更精细的特征工程 + 集成多模型投票 |
| 你们的预测和实际结果对比如何？ | 比赛结束后的实时更新结果 |

## 评分映射

| 评分项 | 权重 | 对应幻灯片 | 关键信息 |
|--------|------|-----------|---------|
| 代码质量 30% | Slide 3 | 架构设计 + 代码组织 — 展示 GitHub 仓库结构 |
| 路演 30% | Slide 1-6 | 表达清晰 + 演示流畅 + 回答自信 |
| 命中率 40% | Slide 5 | 预测结果 — 尽量诚实地展示命中率 |
